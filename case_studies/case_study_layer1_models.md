# Case Study: Layer 1 — Database & ORM Models

## What We Built

Two SQLAlchemy ORM tables: `procurement_requests` (the core record) and `audit_log` (field-level history). Plus the database engine factory with environment-aware connection strings and a `get_db()` dependency for FastAPI's injection system.

**Files:**
- `backend/database.py` — engine, session factory, `init_db()`, `get_db()`
- `backend/models.py` — `ProcurementRequestORM`, `AuditLogORM`
- `tests/conftest.py` — in-memory SQLite fixture
- `tests/test_models.py` — 9 tests, all passing

**Bug fixed during this layer**: `AuditLogORM.request_id` was missing the `ForeignKey("procurement_requests.id")` constraint. SQLAlchemy 2.0 requires an explicit FK or `primaryjoin` expression to resolve relationships — it won't infer across String columns by name. The fix was one import and one argument.

## How It Mirrors Omnea

Omnea's data model is built around a central **Request** object (they call it a "procurement request" or "intake request") that accumulates state as it moves through the workflow. Every field we defined has a direct counterpart in Omnea's UI:

| Our Field | Omnea UI Label | Notes |
|---|---|---|
| `supplier_name` | "Supplier" | Omnea enforces "supplier" not "vendor" |
| `spend_amount` | "Contract Value" | Shown in the intake form |
| `spend_type` | "Payment Type" | one-time / recurring / subscription |
| `category` | "Category" | Software, Hardware, Services, etc. |
| `data_access` | "Data Classification" | Drives TPRM questionnaire depth |
| `questionnaire_depth` | Not visible to requester | Internal — basic/standard/deep |
| `required_approvers` | Shown in "Approval Chain" | Dynamic list, not hardcoded |
| `policy_flags` | Shown as inline warnings | "Finance approval required", etc. |
| `cost_center` | "Cost Center" | Budget coding for Finance |
| `is_new_supplier` | "New Supplier?" toggle | Triggers enhanced due diligence |
| `status` | Kanban stage | pending → approved/rejected |

The **`AuditLogORM`** directly mirrors Omnea's audit trail feature. Every status change, field edit, and approval action is logged with who did it and when. This is critical for compliance — procurement teams need to prove to auditors that the right people approved the right things at the right time. Omnea shows this as a timeline in the request detail view.

**`questionnaire_depth`** is Omnea's most important non-obvious field. It's invisible to the requester but controls how deep the Supplier Portal questionnaire goes. A £500 SaaS tool gets "basic" (5 questions). A £500k data processor handling personal data gets "deep_due_diligence" (40+ questions about security controls, certifications, sub-processors). The policy engine sets this field; the supplier portal reads it.

## Benefits

**Explicit schema = no silent surprises.** Every field is typed, constrained, and tested. A missing `ForeignKey` declaration was caught in 0.05 seconds by the test suite — not in production when a status change causes a cascade delete to fail silently.

**JSON columns for dynamic lists.** `required_approvers` and `policy_flags` are `JSON` columns (SQLite stores them as text, Postgres stores them natively). This means the policy engine can return a variable number of approvers without us needing a separate `approvers` table. The tradeoff: you can't SQL-query individual approvers efficiently. For a POC where we're fetching the whole request anyway, this is the right call.

**Audit log as a first-class table.** Many systems bolt on audit logging as an afterthought. By designing it in from Layer 1, every API endpoint that changes state will write to `audit_log` — no retrofitting needed.

**Environment-aware DATABASE_URL.** `os.getenv("DATABASE_URL", "sqlite:///./procurement.db")` means the same code runs locally on SQLite and on Render/Railway/Heroku with Postgres — no code changes, just an environment variable.

## Potential Pitfalls

**`JSON` columns in SQLite aren't validated.** If you accidentally store `"finance"` (a string) instead of `["finance"]` (a list) in `required_approvers`, SQLite won't complain — it'll just store the string. The downstream code that tries to iterate the list will crash at runtime. Fix: validate in the service layer before writing, never trust that the column contains what you expect.

**`default=_new_id` runs at Python level, not DB level.** The UUID is generated by Python, not the database engine. This means if you insert rows via raw SQL (e.g., a seed script), the UUID won't be auto-generated. Not a problem for our use case, but worth knowing.

**`onupdate=_now` on `updated_at` only fires on SQLAlchemy UPDATE calls.** If you update a row using raw SQL or a bulk update, `updated_at` won't change. Same caveat — fine for our API-driven system.

**Cascade `all, delete-orphan` on audit_logs.** If a procurement request is deleted, all its audit logs are also deleted. This is the right behavior for a demo (clean slate). In a real compliance context, audit logs must be immutable and retained even after the parent record is archived — you'd want to archive, not delete, requests.

**No index on `supplier_name`.** The current schema has no index on supplier fields. Duplicate supplier detection (checking if `Figma` already exists) would do a full table scan. At 10,000 suppliers, this becomes slow. Fix: `index=True` on `supplier_name` or a GIN index in Postgres for fuzzy matching.

## Areas for Improvement

1. **Add Alembic migrations** — `create_all()` can't handle column additions or renames. Once you have real data, you need proper versioned migrations.
2. **Add `archived` boolean** instead of deleting requests — for compliance, records should be immutable.
3. **Add `company_id` index** — for multi-tenant queries (all requests for company X).
4. **Add `supplier_name` index** — for duplicate detection queries.
5. **Separate `ApproverORM` table** — if you ever need to query "all requests awaiting finance approval", a JSON column isn't queryable. A proper join table would be needed.

## Logic Flow

```
Application startup
  └── init_db() called
       └── imports backend.models (side effect: registers ORM classes with Base.metadata)
       └── Base.metadata.create_all(engine)
           └── Emits:
               CREATE TABLE IF NOT EXISTS procurement_requests (id TEXT PRIMARY KEY, ...)
               CREATE TABLE IF NOT EXISTS audit_log (id TEXT PRIMARY KEY, request_id TEXT REFERENCES procurement_requests(id), ...)

API request arrives at POST /chat
  └── FastAPI dependency: Depends(get_db)
       └── get_db() yields a Session
       └── Handler runs, uses session to read/write
       └── get_db() finally: session.close()
```

## Code Flow

`database.py` is the composition root for persistence:
- `engine` — the SQLAlchemy engine, created once at module import time
- `SessionLocal` — a session factory (not a session itself), called per-request
- `Base` — the declarative base that all ORM classes inherit from
- `init_db()` — called once at startup; imports `models` as a side effect to register classes, then calls `create_all`
- `get_db()` — a generator used with `Depends(get_db)` in FastAPI endpoints; yields a session and closes it in `finally`

`models.py` defines two ORM classes:
- `ProcurementRequestORM` — 23 columns. The `JSON` columns (`required_approvers`, `policy_flags`) are stored as SQLite text and deserialized automatically by SQLAlchemy's JSON type.
- `AuditLogORM` — 8 columns. The `ForeignKey("procurement_requests.id")` on `request_id` enables the `relationship()` and allows SQLAlchemy to auto-join without manual SQL.

The relationship is bidirectional:
- `ProcurementRequestORM.audit_logs` → list of `AuditLogORM` (one-to-many, cascade delete)
- `AuditLogORM.request` → single `ProcurementRequestORM` (many-to-one)

Both sides use `back_populates` to stay in sync within the same session.
